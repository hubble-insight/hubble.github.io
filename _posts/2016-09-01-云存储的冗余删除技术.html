云存储中的重复删除技术

1. motivation：

云存储对存储容量的要求是非常大的，对于一个上了规模的云存储系统（一百万左右的用户），如果向每个用户提供1T的容量（像金山公司推出的金山t盘，允许用户使用1T的容量），如果不采用冗余数据删除的技术，那么系统总规模将达到E级。如今虽然数据总量在爆炸似的增长，可对于个人来说，其实自己产生的数据是很少的，主要是疯狂的copy他人产生的数据：像电影，音乐，图片，书籍，软件，文档。其中越是热门的文件，copy的量也是越多（Since more and more files are being copied across file systems rather than generated locally, we can expect identifying and coalescing identical copies to become increasingly important in systems that aggregate file systems. Examples of systems with such support are the FARSITE distributed file system , the Pastiche peer-to-peer backup system [8], and the Single Instance Store in Windows file servers [5].这段来自A Five-Year Study of File-System Metadata论文）。因此云存储中的冗余删除技术是很有价值的。但现在冗余删除技术主要用在备份系统中，由于备份系统和与云存储的在很多方面是不同（下面会详细介绍），所以现有的在备份系统中的冗余删除技术不能直接利用。本文提出了一种适合在云存储中的重复数据删除技术。

 

2. background：

重复数据删除是最近这几年出现的新的技术。它的基本思想是尽可能减少数据中的冗余数据，从而达到减少存储的总量的目的。

2.1  重复数据删除的分类

重复数据删除可以从不同角度分成不同的类，下面列出主要的分类：

A. 按重复数据删除的颗粒分类

由于从不同层次进行重复数据删除，其冗余数据所指也不同。现在冗余数据删除可在文件级、块级、字节级。现有的系统主要是在文件级（Single instance storage in windows）和块级（venti）。在文件级这个层次，冗余数据是指数据完全一样的文件；而在块级层次，冗余数据是指这些文件所共享的那些数据块。

B. 按重复数据删除地方分类

重复数据删除可以在用户端进行（也称前端），在用户端进行的优点是可以减少不必要的数据的传输，缺点是在需要在传输前，判断哪些数据需要传输，哪些则不需要；重复数据也可以在服务器端进行（也称后端），其优点是传输数据之前不需要进行重复数据的判断，直接传到服务端，然后让服务端进行重复数据删除，这样就避开了在存储文件时查找指纹的瓶颈，缺点是需要传输所有数据。

C. 按重复数据删除的精确度分类

精确重复数据删除和不精确重复数据删除，这主要因为随着数据规模的增大，一般都会采用多个存储结点来存储数据，如果允许多个存储结点间可以有公共的数据块，那么这种就是不精确的重复数据删除；如果不允许，则是精确重复数据删除。

2.2 重复数据删除的研究方向

A. 分块算法

重复数据删除按照消重的粒度可以分为文件级和数据块级。文件级的重复数据删除技术也称为单一实例存储(SIS, Single Instance Store)，数据块级的重复数据删除其消重粒度更小，可以达到4-24KB之间。显然，数据块级的可以提供更高的数据消重率，因此目前主流的重复数据删除产品都是数据块级的。数据分块算法主要有三种，即定长切分(fixed-size partition)、CDC切分(content-defined chunking)和滑动块(sliding block)切分。

定长分块算法采用预先义好的块大小对文件进行切分，并进行弱校验值和md5强校验值。弱校验值主要是为了提升差异编码的性能，先计算弱校验值并进行hash查找，如果发现则计算md5强校验值并作进一步hash查找。由于弱校验值计算量要比md5小很多，因此可以有效提高编码性能。定长分块算法的优点是简单、性能高，但它对数据插入和删除非常敏感，处理十分低效，不能根据内容变化作调整和优化。

CDC(content-defined chunking)算法是一种变长分块算法，它应用数据指纹(如Rabin指纹)将文件分割成长度大小不等的分块策略。与定长分块算法不同，它是基于文件内容进行数据块切分的，因此数据块大小是可变化的。算法执行过程中，CDC使用一个固定大小(如48字节)的滑动窗口对文件数据计算数据指纹。如果指纹满足某个条件，如当它的值模特定的整数等于预先设定的数时，则把窗口位置作为块的边界。CDC算法可能会出现病态现象，即指纹条件不能满足，块边界不能确定，导致数据块过大。实现中可以对数据块的大小进行限定，设定上下限，解决这种问题。CDC算法对文件内容变化不敏感，插入或删除数据只会影响到检少的数据块，其余数据块不受影响。CDC算法也是有缺陷的，数据块大小的确定比较困难，粒度太细则开销太大，粒度过粗则重复数据删除效果不佳。如何两者之间权衡折衷，这是一个难点（The low band width network file system首先提出CDC）。

滑动块(sliding block)算法结合了定长切分和CDC切分的优点，块大小固定。它对定长数据块先计算弱校验值，如果匹配则再计算md5强校验值，两者都匹配则认为是一个数据块边界。该数据块前面的数据碎片也是一个数据块，它是不定长的。如果滑动窗口移过一个块大小的距离仍无法匹配，则也认定为一个数据块边界。滑动块算法对插入和删除问题处理非常高效，并且能够检测到比CDC更多的冗余数据，它的不足是容易产生数据碎片。（Duplicate Management for Reference Data首先提出滑动块发）。

3. 云存储中的重复数据删除技术

由于云存储与备份系统的在很多方面是不同，所以现有的在备份系统中的重复数据删除技术直接在云存储中使用是不适宜的。

3.1 重复数据删除技术在云存储和备份中的区别

A. 文件集

云存储中，个人用户一般存储是对自己直接有价值的文件，像多媒体文件，文档文件等，一般不会存储系统文件，而备份系统可能对某特定文件的备份，也可能对某个目录的所有文件备份，因此面临的情况要复杂些。而在云存储中，主要是多媒体和文档类的文件，多媒体文件的特点是，每个文件一般都很大，但是一般不进行修改，且多媒体的文件数却不是很大，而文档文件，每个文件一般很小，但一般会也只有小部分进行修改，且文档文件的数是很大的，因此本文设计的系统对这种特点进行了优化。（局部空间和全局空间）

B. 操作

在备份系统中，一般针对文件集进行操作。一般有的操作是：全备份，增量备份，差异备份。而在云存储中，我们操作的单位是文件，当然我们也可以在云存储的基础上提高备份服务，如Cumulus: Filesystem Backup to the Cloud就是在亚马逊提供的云存储上提高备份服务。我们的云存储的操作单位应该是文件，而不是文件集，正因为这种操作不同，让云存储失去了备份系统中的指纹的局部性。（局部空间增加局部性）

C. 局部性

这里的局部性是指指纹的局部性：上次备份时数据块的指纹出现的序列和这次备份出现的序列相似。DDFS [5] and Sparse Indexing[6]就是利用了这种局部性大大提高了指纹索引。但在云存储中，就没有了这种局部性，因此指纹的索引将是个大问题。

（DDFS [5] and Sparse Indexing[6], proposed novel schemes to effectively eliminate the duplicate-lookup disk bottleneck by exploiting chunklocalities in disk-to-disk (D2D) backup streams）

D. Restore 的性能要求

在备份系统中，文件恢复的速度要求不是很高的，因为在备份系统，文件恢复事件不是经常事件，快速的备份才是经常事件，因此备份系统在备份速度方面要求很高；而在云存储中，文件存储和文件访问都是经常事件，都需要优化，都需要提高各自的速度。

3.2 云存储中使用重复数据删除是否合理

Dedupe技术目前大量应用于数据备份与归档系统，因为对数据进行多次备份后，存在大量重复数据，非常适合这种技术。事实上，dedupe技术可以用于很多场合，包括在线数据、近线数据、离线数据存储系统，可以在文件系统、卷管理器、NAS、SAN中实施。Dedupe也可以用于数据容灾、数据传输与同步，作为一种数据压缩技术可用于数据打包。为什么dedupe技术目前主要应用于数据备份领域，而其他领域应用少呢？这主要由两方面的原因决定的，一是数据备份应用数据重复率高，非常适合dedupe技术。二是dedupe技术的缺陷，主要是数据安全、性能。dedupe使用hash指纹来识别相同数据，存在产生数据碰撞并破坏数据的可能性。dedupe需要进行数据块切分、数据块指纹计算和数据块检索，消耗可观的系统资源，对存储系统性能产生影响。信息呈现的指数级增长方式给存储容量带来巨大的压力，而dedupe是最为行之有效的解决方案，因此固然其有一定的不足，它大行其道的技术趋势无法改变。更低碰撞概率的hash函数、多核、GPU、SSD等，这些技术推动dedupe走向成熟，由作为一种产品而转向作为一种功能，逐渐应用到近线和在线存储系统。ZFS已经原生地支持dedupe技术，我们相信将会不断有更多的文件系统、存储系统支持这一功能。

我认为下面的一个原因可以认为在云存储系统中使用重复删除技术是必要的。

1) 存储容量巨大

这点很明显，如果某个企业向每个用户提供100G的容量，用户为1000万（像金山公司推出的金山t盘，允许用户使用1T的容量），如果不采用冗余数据删除的技术，那么系统总规模将达到将近E级。当然这里有个疑惑：到底使用重复数据删除效果好不好，它带来的利是否大于带来的弊？不过从直观上来讲,用户存储的数据一般为：多媒体文件和文档文件，而多媒体文件是文件级重复数据删除重点。对于文档文件，大部分也是用户通过copy得来的，而自己产生的也只有小部分，因此对于文档文件，我们把那些不怎么写的，做文件级重复数据删除，经常写的做块级重删（怎样预测哪些文件是经常写的是系统的一个特点）。通过这种策略，系统中的重复数据删除效果应该会不错的。同时重复数据删除技术带来的“读问题”和“数据可靠性”可以通过把文件尽量分成大块和块的副本机制来缓解（为什么文件级重复数据删除的文件也需要分块？引入块的副本机制是否与重复数据删除矛盾？下面有解释），

2) 数据可靠性

目前，数据的可靠性主要通过“数据副本”和“纠错码”两种途径解决。如果云存储中没用使用重复数据删除技术，现在假设下面一个情景：有二十个用户，各存有“my love.mp3”，为了尽大可能保证该文件的可靠性，那么需要对这同一个文件的不同份进行可靠性保护，显然代价很大。如果使用了重复数据删除，那么这二十个用户共享一个“my love.mp3”文件，那么只需对这一份进行“数据副本”货“纠错码”保护。

3) 资源的利用率

个人数据在快速增长，但有个事实确实：我们在一大堆个人数据中，其实我们通常只是用了使用其中很少部分的数据，绝大部分的个人数据闲置在一边，而通过重复数据删除技术，就可以提高资源的利用率，因为你不用的时候，别人可能在使用。

4. Inspiration of设计

这里有个前提假设:用户在线存储的文件主要分为两大类，按文件类型分：1.多媒体文件，像视频文件，图片文件，音频文件等；2.文档文件，像.doc 、.pdf、.ppt、.txt等。对于多媒体文件一般是大文件，量不多，但占的存储容量很大，且一般不修改，对于文档文件，量多，但总的占存储容量不大，且一般也只有小部分会修改。按另一种方式分类：1不修改的文件，2会修改的文件。通过第一种分类，我们知道绝大部分多媒体文件和大部分文档文件是不修改的，只有极小部分多媒体文件，和小部分文档文件是要修改的。

其实这个前提假设也是很直观的。至于哪种文件数量或容量占得比例并不重要，重要的是这两大类文件的“写”的情况，我们知道在重复数据删除的技术中，重复数据删除比例和其所进行的数据集是有很大关系的，一般对于多媒体文件，块级的重复数据删除效果很不好，当然除非多媒体文件中有很多文件级的重复文件；对于文档文件，一般是分块颗粒越小，重复删除比例就越高，尤其是对于相近的文档文件（一般是一个文件进行修改得到的多个版本的文件），效果甚佳。

目前重复数据遇到的问题是指纹索引而引起的扩展性、读性能和数据可靠性（计算瓶颈在备份系统可能会存在，但是在云存储中不会存在，因为备份系统需要高速备份，在较短时间内，需要计算大量文件的指纹，但在云存储中，只需计算用户需要存储的文件，因此计算量和紧迫程度都不大）。在现有的带有重复数据删除的备份系统中，一般把数据分成不定长的、小的（在4k左右）的数据块，然后只存储不同的数据块，因此我们需要判别需要存储的新的数据块是否已经存在了。现有的技术一般是计算每个数据块的一个hash值（也称为该块的指纹，因为我们可以认为不同hash的数据块，数据块本身也不同），然后就看需要存的数据块的指纹是否在已经存的数据块的指纹集中存在。这个过程看上去很简单，但是有个问题，就是已经存的数据块的指纹集会很大，一般会超过内存的容量。这样查找每个指纹都可能需要访问硬盘，这效率是非常低的。解决这个问题的一个有效办法是利用指纹的局部性，增加命中，减少访问硬盘。但在云存储中，指纹的局部性又因为文件操作单位和备份系统的不同而大大破坏，因此想在云存储系统中，利用局部性很难。至于是否可以利用文件的相似性，我认为在云存储中，用户间的文件要不相同，要不就是相似度极底，对于用户内的文件，也只有那些把一个文件进行修改得到的多个版本的文件间相似度很高。于是自己根据自己的假设，在客服端把文件分层两大类：1：一般不会进行写的文件、2：可能会进行写的文件。对第一类，我们利用分布式技术，把数据（数据按大块分块,按定长分块还是按滑动块待考虑）按照自己的相应指纹 均匀的路由到一个含有大量的pc集群中，这样使得每台pc含的数据指纹都在内存里（因为是大块，所以指纹都可以放在内存中），且这部分文件是在全局空间进行重复数据删除，在MAD2: A Scalable High-Throughput Exact Deduplication Approach for Network Backup Services（杨天明学长）中，也使用分布式hash把数据块均匀的分布在存储群中，但mad2，有两个很大的问题：如果一个3M的文件按平均4k进行不定长分块，假设得到几百块数据块，而这几百块数据会均匀分布在存储结点的，因此当进行读或写时，需要同时访问很多个存储结点，当然这些可以并行进行，但要保证每个存储结点的延迟差不多，且每个存储结点没有发生故障，且那时刻意味着存储结点集群中很多点都在为它服务，可以看这种模式的操作很没有效率，且数据的可靠性差，读性能也不好；还有个问题是不知道杨学长用的是哪种分布式hash，分布式hash一般是logN级别的路由效率，在p2p系统中，这种路由效率还是可以接受，但在数据中心中等要求性能很高的地方，就可能不能接受了，尤其当存储结点很多时。对于可能写的文件，一般是小文件，我们把这些文件都放在一个结点，按内容进行分块，在这个结点上进行局部的重复数据删除。

我们可以把一般不修改的文件（绝大部分多媒体文件，和大部分文档文件）按大块（如几兆，不同文件大小级别的使用不同长度的定长的分块）进行定长分块（因为这些文件一般不进行修改，所以采用定长分块），这样一方面减少数据块指纹，同时也减少了一个文件分布在不同存储结点的数量；然后使用RUSH算法（保证在常数的路由效率，并且支持结点的删除、添加和副本机制），把这些大块路由到相应的存储结点。对于小部分的，会修改的文档文件，我们把这些文档文件路由到一个存储结点上去，按不定长分块。对于大块的重复数据删除，我们在全局空间进行，而对于那些修改的文档文件，我们就在这些文档所存在的那个存储结点上进行局部的重复数据删除，这种混合删除具体分析可见SAM: A Semantic-Aware Multi-Tiered Source De-duplication Framework for Cloud Backup（谭玉娟学姐）。

简单来说就是：根据文件写的情况，把文件分类，对不同类的文件采用不同策略进行重复数据删除，以减少“系统开销”,提高“读性能”，同时采用数据块副本技术来提高“数据可靠性”。（也是自己的主要思想）

下面有几个必须回答的问题：

1) 怎么对待重复数据带来的系统开销

重复数据删除需要进行数据块切分、数据块指纹计算和数据块检索，消耗可观的系统资源，对存储系统性能产生影响。因此我们一定要把这些开销限制在一定范围内，在该系统中，我们主要利用“绝大部分文件不修改”这点，把这些不修改的文件（总文件的很大部分）按大块进行分块，如果平均每块大小为1M，每块指纹为20byte，那么对于对于存储容量为1t的pc机，就算1t容量用完，其指纹大小20M，这完全可以放在内存中。

2) 怎样提高读性能

由于我们很大部分文件是大块分块的，相对小块分块的读性能要好的多。

3) 数据的可靠性

对于需要高可靠的数据，我们使用副本机制来保证数据的可靠性。

4) 计算瓶颈

由于数据块切分、数据块指纹计算是分布在各个用户上的，所以计算不会成为瓶颈

设计的目标可从两方面来说：1对于用户，上传和下载文件的速度要快（这也是为什么，即使在文件级的重复数据删除，也把该文件分块，再存在不同结点上，这样可以并行上传或下载），数据要安全可靠；2对于服务提供者，需要减少系统总容量，减少成本，且系统需要一种自我进化的功能（淘汰旧的，性能不好的机器，增加新的，性能好的机器，且这个过程不影响用户使用）。

技术指标如下：

l 能够提供P级甚至E级的总存储容量

l 能够支持高速下载和上传

l 能够保证数据的可靠性，安全性，一致性。 

l 系统能够自我进化
